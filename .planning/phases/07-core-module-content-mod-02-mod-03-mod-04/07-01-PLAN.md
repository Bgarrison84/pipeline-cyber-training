q-01 (complianceControls: ["NIST-SC-7"]):
stem: "Which PowerShell cmdlet audits currently enabled inbound firewall rules on a Windows system?"
a: "Get-NetAdapter" (incorrect — Get-NetAdapter lists network interfaces, not firewall rules)
b: "Get-NetFirewallRule -Enabled True" (correct — retrieves active rules from Windows Firewall; filter with Direction Inbound to focus on inbound policy)

d: "Test-NetConnection" (incorrect — tests connectivity to a target host; does not enumerate rules)
explanation: "Get-NetFirewallRule -Enabled True retrieves all enabled Windows Firewall rules. Pipe to Where-Object {$_.Direction -eq 'Inbound'} to focus on inbound policy relevant to SC-7 boundary protection."

q-02 (complianceControls: ["TSA-NetworkSeg"]):
stem: "Your automated scan finds TCP port 3389 (RDP) open inbound on a pipeline boundary firewall. Before blocking it, what should you do first?"
a: "Immediately run New-NetFirewallRule to block it" (incorrect — blocking without investigation could disrupt an authorized maintenance window already in progress; investigate first to determine if it is authorized)
b: "Check the firewall change log and verify with the change manager whether an authorized change window is open for this port" (correct — port 3389 may be open for a legitimate, time-limited maintenance window; verify authorization before blocking to avoid disrupting sanctioned activity while still complying with the TSA network segmentation mandate)
c: "Reset the server to close the connection automatically" (incorrect — server reset is a disruptive action that destroys evidence of who opened the port and why)
d: "Log the event and take no further action until the next scheduled audit" (incorrect — an open RDP inbound rule on a boundary device must be addressed promptly; deferring to the next audit cycle violates TSA incident response expectations)
explanation: "The TSA network segmentation mandate requires prompt action on unauthorized access paths, but 'investigate first' is always the correct initial step — you need to determine whether an authorized maintenance window explains the open port before taking containment action."

q-03 (complianceControls: ["NIST-SC-7", "TSA-NetworkSeg"]):
stem: "Which PowerShell command maps a listening TCP port to the process that opened it?"
a: "Get-Process -Name svchost" (incorrect — Get-Process retrieves processes by name but does not show which port a process is listening on)
b: "netstat -ano" (incorrect — netstat shows port-to-PID mapping but is not a PowerShell cmdlet; use Get-NetTCPConnection for a PS 5.1 approach)
c: "Get-NetTCPConnection -State Listen | Select-Object LocalPort, OwningProcess" (correct — returns a table of listening ports with the PID of the owning process; use Get-Process -Id on the PID to get the process name)
d: "Test-NetConnection -Port 445" (incorrect — Test-NetConnection tests outbound connectivity to a remote port; it does not enumerate local listening ports)
explanation: "Get-NetTCPConnection -State Listen returns all locally listening TCP ports with OwningProcess (PID). Follow up with Get-Process -Id <PID> to identify the process name — this two-step approach satisfies NIST SC-7 boundary protection auditing requirements."
