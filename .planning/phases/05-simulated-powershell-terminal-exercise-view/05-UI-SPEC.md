## Interaction & States

### I-01: Command Input

- Enter key on input: trim whitespace, if empty do nothing; if non-empty: echo command to terminal output, pass to command handler, clear input
- ↑ arrow key: cycle backward through command history array (session-scoped, not persisted)
- ↓ arrow key: cycle forward through command history (restores partial draft if at end of history)
- Click anywhere in terminal body (not on input): focus the input element
- Input autofocus: on exercise load, input receives focus
- Tab key: no-op (no autocomplete in Phase 5)
- All other keys: default browser behavior (character input)

### I-02: Command Matching Sequence

Per D-09 and D-10, the handler runs this order for each submitted command:

1. Test trimmed input against each `expectedCommands[].pattern` (RegExp, case-insensitive by default)
2. If match found: print `successOutput` to terminal (plain text, white-space pre-wrap), advance step
3. If no match: test against each `hintPatterns[].pattern`
4. If hint match found: display that entry's `hint` text in the step panel hint area (does not print to terminal)
5. If no hint match: print `feedbackOnWrong` text into terminal output in `var(--color-text-muted)` color

Unrecognized commands (no pattern match + no hint match + feedbackOnWrong):
- Terminal output line: `feedbackOnWrong` text in var(--color-text-muted)
- No JS exception thrown — all RegExp construction wrapped in try/catch

### I-03: Step Advancement

- On step success: step card gets left-border #22c55e, icon swaps to check-circle, instruction text dims to var(--color-text-muted)
- Next step card becomes active: icon shows "terminal" in var(--color-accent), instruction text is var(--color-text-primary), hint area hidden
- Step counter heading updates: "Step 2 of 3"
- activateIcons() called after DOM mutation to render new Lucide icons

### I-04: Exercise Completion

- Triggered when last step succeeds
- `progressStore.saveExercise(moduleId, exerciseId)` called immediately
- `progressStore.markLessonCompleted(moduleId, lessonId)` called (lessonId derived from exerciseId or passed from modules-config)
- `import('./sidebar.js').then(m => m.refreshSidebarProgress(moduleId))` — dynamic import (circular dep avoidance)
- Terminal input: readonly set, prompt row opacity 0.4
- Completion banner (C-06) injected below terminal element
- No redirect — user navigates away manually

### I-05: Re-visit State (exercise already completed)

- `progressStore.getExerciseCompletion(moduleId, exerciseId)` returns `{completed: true, completedAt: '...'}|null` on load
- If completed: render all steps in done state (check-circle, muted instruction), terminal disabled, completion banner shown immediately
- Completion banner row 1 variant for revisit: "Exercise previously completed — {date slice YYYY-MM-DD}"
- User can still read all step instructions; they cannot re-enter commands (terminal is readonly)

### I-06: Hint Display

- Hint text appears in the step panel hint area (C-03 hint sub-component), NOT in the terminal output
- Hint area transitions from `display:none` to `display:block` — no animation required (Phase 5 simplicity)
- Hint text replaces previous hint text on each new near-miss (only latest hint shown)
- Hint area is cleared (hidden) when step advances

### I-07: Terminal Scroll Behavior

- Each new output line: `terminalBody.scrollTop = terminalBody.scrollHeight` after appendChild
- Terminal body has `overflow-y: auto` and `max-height: 480px`
- User can scroll up to review history; new output auto-scrolls to bottom

### I-08: Focus Ring

- All interactive elements use the global `*:focus-visible` rule from `src/style.css`:
  `outline: 3px solid #f97316; outline-offset: 2px`
- Terminal input: focus ring suppressed inside terminal (outline none on the input, terminal border serves as the focus indicator — the terminal border color shifts to #22c55e when input is focused)
  - Terminal wrapper border: `var(--color-border)` default → `#22c55e` on input focus → `var(--color-border)` on blur

### I-09: Storage Unavailability Warning

- Pattern mirrors lesson-view.js storage warning
- If `!progressStore.isStorageAvailable()`: prepend warning paragraph to exercise view column
- Copy: "Progress cannot be saved — storage unavailable. Your exercise completion will not be recorded."
- Style: var(--text-body), var(--color-text-muted), role="alert"

## Security Constraints (carry-forward from prior phases)

- `esc()` from `src/utils/escape.js` applied to ALL exercise JSON strings before `innerHTML` insertion: `title`, `description`, `context`, `instruction`, `hint`, `successOutput`, `feedbackOnWrong`, `hintPatterns[].hint`
- `moduleId` and `exerciseId` values from URL hash are never injected into innerHTML text content — only into safe attribute positions (`href`, `data-*`) after `esc()` wrapping
- Terminal input value is passed to RegExp matching only — never injected into DOM as innerHTML
- `successOutput` from exercise JSON displayed via `textContent` assignment or `esc()` — never raw innerHTML (exercise authors may include angle brackets in PS output strings)
