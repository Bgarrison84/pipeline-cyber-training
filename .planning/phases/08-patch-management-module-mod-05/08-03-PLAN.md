title: "Critical CVE vs. OT Vendor Qualification Window"
complianceControls: ["TSA-PatchMgmt", "NIST-SI-2"]

narrative: Set the scene — a critical CVE targeting Windows Server 2019 has been published. PIPELINE-HIST01 (the pipeline data historian, running Windows Server 2019) is confirmed affected by the CVE. Microsoft's patch is available immediately in the Windows Update catalog. The OEM vendor for the historian software has not yet tested or qualified this patch — their qualification process typically takes 3-6 months. Pipeline operations are currently active (no scheduled maintenance window for 6 weeks).

"In OT environments:" paragraph in narrative: a Windows Server patch that is unqualified by the OEM can cause the historian's OPC server or data acquisition service to crash or corrupt data; during active pipeline operations, a historian outage means operators lose real-time process data visibility — a safety-impacting event that may require a manual operational response; the risk of an unqualified patch during operations outweighs the risk of operating with a compensating control while awaiting OEM qualification.

Phase 1 ("Discovery") — title: "Discovery", isFinal: false, both options nextPhaseId: "phase-2":
prompt: "A critical CVE targeting Windows Server 2019 is confirmed to affect PIPELINE-HIST01. Microsoft's patch is available immediately. The historian OEM has not yet qualified this patch. What is the correct first response?"

opt-a (incorrect): text: "Apply the Microsoft patch to PIPELINE-HIST01 immediately — the CVE severity justifies bypassing OEM qualification."
outcome: "Applying an unqualified patch to PIPELINE-HIST01 during active operations risks crashing the OPC server or data acquisition service. If historian data collection stops, operators lose visibility into real-time process measurements — potentially a safety event requiring manual operational response. The TSA patch management mandate for OT systems requires OEM qualification before patching; CVE severity does not override this requirement for active OT systems. Document the CVE and open a vendor qualification request immediately."

opt-b (correct): text: "Implement compensating controls (network isolation for PIPELINE-HIST01, enhanced monitoring on any new connections) and contact the OEM vendor to request patch qualification status."
outcome: "Correct. For OT systems, compensating controls + vendor qualification request is the required response when a patch is not yet OEM-qualified. Network isolation — blocking all inbound connections to PIPELINE-HIST01 except required historian polling ports — reduces exposure to the CVE while the qualification process runs. Document the compensating controls with: the CVE ID, the vulnerability it mitigates, the date implemented, your name as approver, and the review interval. This documentation is what a TSA auditor will request."

Phase 2 ("Aftermath") — title: "Aftermath", isFinal: true, both options nextPhaseId: null:
prompt: "Three days later, an investigation reveals that IT's WSUS server auto-pushed the patch to an OT workstation connected to the SCADA system — outside any change window — because the OT workstation was inadvertently in the IT WSUS target group. The patch caused a PLC polling service crash on the workstation."

opt-a (incorrect): text: "Quietly fix the OT workstation, roll back the patch, and log the incident in the internal IT ticketing system only."
outcome: "This response has two critical failures: first, quietly fixing the workstation without TSA notification violates the current TSA pipeline security directive — unauthorized changes that affect OT operations trigger mandatory incident reporting; second, the root cause (OT workstation in the wrong WSUS target group) remains unaddressed, making another auto-push incident certain. TSA auditors reviewing the ticketing record will find a known incident with no notification and no root cause remediation — a compounding finding."

opt-b (correct): text: "Notify TSA per the incident reporting requirements, document the incident and root cause, and implement WSUS segmentation to move all OT workstations to a dedicated OT WSUS target group with manual approval only."
outcome: "Correct. TSA notification is mandatory for unauthorized changes that affect OT operations. Documenting the root cause — IT WSUS targeting includes OT workstations — is required for the incident record. Implementing WSUS segmentation (dedicated OT group, manual approval, no auto-deployment to the 10.0.0.0/24 segment) is the architectural fix that prevents recurrence. This response satisfies both the immediate TSA reporting requirement and the longer-term risk reduction mandate."

id: "patch-management-scenario-02"
moduleId: "patch-management"
title: "Maintenance Window vs. Ops Manager Pressure"
complianceControls: ["TSA-PatchMgmt", "NIST-MA-2"]

narrative: The OEM vendor has completed qualification of the CVE patch for PIPELINE-HIST01. This is 4 months after the CVE was published — within the typical 3-6 month OEM qualification window. The next scheduled maintenance window for PIPELINE-HIST01 is 3 weeks away. The operations manager has requested that the team skip the scheduled window and patch PIPELINE-HIST01 now to close the CVE faster and demonstrate responsiveness to the TSA.

"In OT environments:" paragraph in narrative: scheduled maintenance windows for OT systems exist not merely for scheduling convenience — they coordinate with OT operations to ensure the historian is in a safe state to restart, that operators are prepared for a potential historian interruption, and that a rollback plan is staged and ready; patching outside a maintenance window removes these safeguards; even with an OEM-qualified patch, an unplanned historian restart can interrupt pipeline process data collection at a critical monitoring moment.

Phase 1 ("Scheduling") — title: "Scheduling", isFinal: false, both options nextPhaseId: "phase-2":
prompt: "The OEM has qualified the CVE patch. The next maintenance window is 3 weeks away. The operations manager wants to patch PIPELINE-HIST01 now to close the CVE faster. What do you do?"

opt-a (incorrect): text: "Skip the scheduled maintenance window and patch PIPELINE-HIST01 immediately to close the CVE."
outcome: "Patching immediately without a scheduled maintenance window bypasses the safety coordination that OT change management exists to provide: operator notification, rollback plan staging, and confirmation that the historian is in a safe state. If the patch fails mid-install and the historian cannot restart, operators are now without process data visibility with no pre-staged rollback plan and no operator standby. The TSA patch management mandate requires controlled maintenance for OT systems — 'faster' is not a compliance argument for bypassing change management."

opt-b (correct): text: "Maintain the scheduled maintenance window timeline; document that the OEM qualification is complete and compensating controls remain in place for the 3-week window period."
outcome: "Correct. Maintaining the scheduled window preserves the safety coordination safeguards: operators are informed, a rollback plan is staged, and the historian is taken offline in a controlled state. Documenting that OEM qualification is complete updates the compensating controls record to reflect reduced risk (the patch is now available and approved) while the controlled window date approaches. This approach satisfies both the TSA change management requirements and the operations manager's accountability need — the patch is scheduled, not deferred indefinitely."

Phase 2 ("Window") — title: "Maintenance Window", isFinal: true, both options nextPhaseId: null:
prompt: "During the scheduled maintenance window, the patch fails to install mid-process. A dialog box requesting a reboot confirmation is frozen on PIPELINE-HIST01's screen. The historian is currently offline (as expected for the window). What do you do?"

opt-a (incorrect): text: "Force-retry the patch installation by clicking through the frozen dialog and restarting the install process without rolling back."
outcome: "Force-retrying a failed mid-install patch on an OT system risks leaving the OS in a partially patched, potentially corrupted state. If the patch installation left some system components updated and others not, the historian software may fail to start — with an unclear root cause. In OT environments, a partial install that corrupts the OPC server or data acquisition service could require a full OS reinstall, which is a significantly longer and more disruptive outage. Roll back first; characterize the failure before retrying."

opt-b (correct): text: "Roll back PIPELINE-HIST01 to its pre-patch clean state, document the failure with the exact error details and the dialog screenshot, and reschedule with OEM vendor support engaged for the retry window."
outcome: "Correct. Rolling back to the last known-good state restores historian service before operations resume at the end of the maintenance window. Documenting the failure — the exact error, the frozen dialog, and a screenshot — is required for the OEM support case that will diagnose the root cause before the retry. Rescheduling with OEM support engaged ensures the retry attempt has expert assistance staged. This response follows the controlled maintenance (NIST MA-2) requirements: every OT maintenance action, including failures, must be documented with outcome records."

id: "patch-management-scenario-03"
moduleId: "patch-management"
title: "TSA Audit: Missing Compensating Controls Documentation"
complianceControls: ["TSA-PatchMgmt", "NIST-SI-2", "NIST-MA-2"]

narrative: A TSA auditor has arrived for a scheduled cybersecurity compliance review of ExampleCorp's pipeline operations. The auditor requests compensating controls documentation for three OT patches that were deferred beyond the standard 30-day IT patch window. Your team has the compensating controls documentation for two of the three deferred patches — the third patch was deferred 8 months ago and no compensating control document was ever created.

"In OT environments:" paragraph in narrative: TSA auditors reviewing compensating controls documentation are specifically looking for three things: evidence the risk was assessed, evidence a compensating measure was implemented, and evidence the documentation was reviewed at defined intervals; a missing document is not a small administrative oversight — it is evidence that the organization may not have had a deliberate risk management process for that deferred patch; the absence of a compensating control document for a security patch affecting OT systems is typically treated as a significant finding.

Phase 1 ("Audit request") — title: "Audit Request", isFinal: false, both options nextPhaseId: "phase-2":
prompt: "The TSA auditor requests compensating controls documentation for 3 deferred OT patches. You have documentation for 2 of them. The third has no documentation. What do you tell the auditor?"

opt-a (incorrect): text: "Tell the auditor the documentation for the third patch is forthcoming and will be provided within the week."
outcome: "Telling a TSA auditor that documentation is 'forthcoming' for a patch deferred 8 months ago signals that the compensating controls process was not followed in real time — a TSA auditor interprets this as a systemic process failure, not a minor gap. Promising future documentation while the auditor is present does not satisfy the audit request; the auditor is reviewing historical compliance, not future intent. TSA may record this as a compliance gap finding and flag the organization for follow-up review."

opt-b (correct): text: "Be transparent with the auditor: present the two documented patches' complete records, acknowledge that the third deferred patch has no compensating controls documentation, and commit to a specific remediation timeline with a corrective action plan."
outcome: "Correct. TSA auditors expect transparency. Presenting the two complete records demonstrates that the organization's compensating controls process works for most cases — the third is an identifiable exception. Acknowledging the gap and committing to a specific corrective action plan (within a defined number of days, with a named responsible person) shows organizational accountability. TSA auditors consistently report that organizations that disclose gaps with clear remediation plans fare significantly better than those who minimize or deny gaps. The gap itself may still be recorded as a finding; the response determines whether it escalates."

Phase 2 ("Remediation") — title: "Remediation", isFinal: true, both options nextPhaseId: null:
prompt: "The audit is complete. The missing compensating controls document is officially a finding. The team must now address the gap for the third deferred patch. How should the documentation be handled?"

opt-a (incorrect): text: "Retroactively reconstruct the compensating controls document based on what the team believes was done 8 months ago, back-date it, and submit it as the original record."
outcome: "Retroactively reconstructing and back-dating documentation is falsification of compliance records — a far more serious violation than the original missing document. A TSA auditor who later discovers back-dated records in a follow-up review will treat it as deliberate misrepresentation, which can result in significantly more severe enforcement action than the original finding. Under no circumstances should compliance documentation be created, back-dated, or altered to misrepresent when it was produced."

opt-b (correct): text: "Implement a formal compensating controls process going forward; document what can be contemporaneously verified (current system state, current network isolation measures) with explicit caveats noting the documentation was created post-audit; include a corrective action plan with a named responsible person and supervisor sign-off."
outcome: "Correct. Post-audit documentation created contemporaneously — with honest caveats noting when it was created and why it was not created at the time of deferral — has more integrity than reconstructed historical records. The corrective action plan with supervisor sign-off demonstrates organizational commitment to preventing recurrence. Submit this package to TSA as part of the finding remediation response. The caveated contemporaneous document acknowledges the failure honestly; that honesty is more defensible in any follow-up review than a falsified historical record."
