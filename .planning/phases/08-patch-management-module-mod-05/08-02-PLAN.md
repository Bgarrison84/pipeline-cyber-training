q-01 (complianceControls: ["NIST-SI-2"]) — covers Windows/IT patching sub-area:
stem: "Which PowerShell cmdlet retrieves a list of all installed Windows patches (hotfixes) on a local system, including installation dates?"
a: "Get-WindowsUpdate" — incorrect — Get-WindowsUpdate is part of the PSWindowsUpdate module which is not installed by default in PS 5.1; use Get-Hotfix for installed patch inventory
b: "Get-Hotfix" — correct — Get-Hotfix returns all installed QFE (Quick-Fix Engineering) patches from the local WMI store; pipe to Select-Object HotFixID, Description, InstalledOn and Sort-Object InstalledOn for an audit-ready inventory; this is the PS 5.1 built-in approach requiring no additional modules
c: "Get-WUHistory" — incorrect — Get-WUHistory is part of the PSWindowsUpdate module; not available by default in PS 5.1
d: "Get-WinEvent -LogName System" — incorrect — Get-WinEvent queries the event log; while patch-related events appear in the System log, it does not return a structured patch inventory list the way Get-Hotfix does
explanation: "Get-Hotfix is the PS 5.1 built-in cmdlet for querying installed Windows patches. It reads from WMI (Win32_QuickFixEngineering) and returns a structured list of HotFixID, Description, InstalledOn, and InstalledBy — all fields required for a compliance audit report."

q-02 (complianceControls: ["TSA-PatchMgmt", "NIST-MA-2"]) — covers OT/ICS patching sub-area:
stem: "A critical CVE has been published for Windows Server 2019. Your PIPELINE-HIST01 historian runs Windows Server 2019. Microsoft's patch is available immediately. What is the correct OT patch management response?"
a: "Apply the Microsoft patch immediately — a critical CVE requires same-day remediation on all systems" — incorrect — applying an unapproved patch to an OT historian without OEM vendor qualification risks corrupting the historian software or OPC server configuration; same-day IT cadences do not apply to OT systems
b: "Contact the historian OEM vendor to request qualification status for the patch; document compensating controls (network isolation, enhanced monitoring) until the vendor qualifies the patch" — correct — OT systems require OEM vendor qualification before patching; the typical qualification period is 3-12 months; during the wait, the organization must document compensating controls (e.g., blocking all inbound connections to PIPELINE-HIST01 except required historian ports, enhanced SIEM monitoring) and get supervisor sign-off; this approach satisfies the current TSA pipeline security directive's OT patch management requirements
c: "Wait for the next scheduled IT patch cycle (30 days) and apply it with standard IT change management" — incorrect — standard IT change management does not satisfy OT patch requirements; OT systems require OEM qualification regardless of the IT patch cycle; applying without OEM qualification after 30 days still violates the OT approval workflow
d: "Reimage PIPELINE-HIST01 to a clean baseline and restore from the most recent backup" — incorrect — reimaging destroys the historian's running configuration and potentially its data archive; this is a last-resort recovery action, not a routine patch management response
explanation: "OT systems (historians, HMIs, PLCs) require OEM vendor qualification before any patch is applied. The typical qualification cycle is 3-12 months. During the wait, the organization must document compensating controls — network isolation, enhanced monitoring, or restricted access — with supervisor approval. This is required by the TSA patch management mandate for OT/ICS environments."

q-03 (complianceControls: ["TSA-PatchMgmt", "NIST-SI-2"]) — covers compliance reporting sub-area:
stem: "A TSA auditor requests patch compliance evidence for your pipeline OT systems. Which combination of PowerShell commands generates the required audit artifact?"
a: "Get-Process | Export-Csv 'C:\\Audit\\processes.csv'" — incorrect — Get-Process exports running processes, not installed patch inventory; this does not satisfy a patch compliance audit request
b: "Get-WinEvent -LogName System | Where-Object {$_.Id -eq 19} | Export-Csv" — incorrect — Event ID 19 (WindowsUpdateClient/OperationalSuccess) shows update installation events but not the full patch inventory; Get-Hotfix is the authoritative inventory source for audit evidence
c: "Get-Hotfix | Export-Csv 'C:\\Audit\\patch-status.csv' -NoTypeInformation" — correct — Get-Hotfix returns the full installed patch inventory and Export-Csv writes it to a structured file the auditor can open in Excel or import into a compliance system; -NoTypeInformation removes the PS type header row that auditors find confusing; C:\\Audit\\ is the established audit evidence staging path for this organization
d: "Invoke-WebRequest -Uri 'https://wsus.examplecorp.local/report'" — incorrect — pulling a WSUS report via HTTP is not a PS command pattern available from the command line; even if it were, OT systems in air-gapped environments (10.0.0.0/24) have no network access to the WSUS server; the local Get-Hotfix approach works on any system regardless of connectivity
explanation: "Get-Hotfix | Export-Csv is the standard PS 5.1 approach for generating patch compliance evidence. The -NoTypeInformation flag removes the PowerShell type header, producing a clean CSV suitable for auditor submission. After exporting, use Get-Content 'C:\\Audit\\patch-status.csv' | Measure-Object -Line to verify the file contains data rows."
