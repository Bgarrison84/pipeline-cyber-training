## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public/data/modules/*.md → browser innerHTML | Lesson Markdown files are fetched and rendered as HTML; files are repo-authored content |
| URL hash params → view renderers | moduleId and lessonId are extracted from the hash and used in fetch URLs and template strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | Frontmatter values (title, lessonId) injected into lesson-view.js innerHTML templates | mitigate | `esc(meta.title)`, `esc(meta.lessonId)` applied to all frontmatter fields before template insertion; renderMarkdown body output is trusted repo content — no esc() on the full HTML blob |
| T-02-05 | Tampering | data-code attribute on copy button receives token.text | mitigate | `esc(token.text)` in content-loader.js code renderer — `rawCode = esc(token.text)` then `data-code="${rawCode}"`. Browser decodes entities when reading `btn.dataset.code` for clipboard write — raw code reaches clipboard correctly |
| T-02-06 | Tampering | moduleId/lessonId from URL hash injected into fetch URLs | accept | Fetch URL construction uses moduleId/lessonId in path only (not innerHTML). A crafted hash would result in a 404 fetch response, handled gracefully. GitHub Pages serves only existing files. |
| T-02-07 | Information Disclosure | Error messages exposing moduleId/lessonId in UI | mitigate | renderLessonNotFound() and renderLessonError() use static text — never inject moduleId or lessonId into the error HTML (mirrors T-03-01 pattern from module-view.js) |
