**OT callout color rationale:** `#d97706` (amber-600) is chosen over `#ca8a04` (yellow-600) because amber-600 reads as a warm informational signal rather than a warning yellow. Yellow-600 sits visually close to standard warning color conventions (risk of misread as "caution/error"). Amber-600 at 8% background opacity is visible but not alarming — appropriate for an informational environmental distinction callout. It is clearly distinct from the main accent orange (`#f97316`) because amber sits in the yellow-orange hue range while the accent is pure orange-red. A trained eye and a colorblind user both see them as different colors.

## Typography

Inherited from Phase 1 (Mono 13px, Body 14px, Heading 20px, Display 28px).

Phase 2 adds prose-specific type roles used ONLY inside the lesson reading column (`.lesson-body`). These do not affect UI chrome.

| Role | Token | Size | Weight | Line Height | Font |
|------|-------|------|--------|-------------|------|
| Prose body | `--text-prose-body` | 16px | 400 (regular) | 1.7 | Inter |
| Prose h2 | `--text-prose-h2` | 20px | 600 (semibold) | 1.3 | Inter |
| Prose h3 | `--text-prose-body` | 16px | 600 (semibold) | 1.4 | Inter |
| Code (in prose) | `--text-mono` | 13px | 400 (regular) | 1.5 | JetBrains Mono |
| Code (block) | `--text-mono` | 13px | 400 (regular) | 1.6 | JetBrains Mono |
| Prose small / caption | `--text-prose-small` | 13px | 400 (regular) | 1.5 | Inter |

**Prose h3 note:** h3 elements use `--text-prose-body` (16px) — no separate size token. Differentiated from prose body exclusively by `font-weight: 600` and `line-height: 1.4`. This avoids introducing a fifth distinct size (17px) that would breach the 4-size limit.

**Combined system font sizes (Phase 1 + Phase 2):** 13px, 14px, 16px, 20px, 28px — five distinct sizes. The 16px prose body and 14px UI body serve genuinely distinct roles (reading prose vs. UI chrome) and justify this one-size exception over the recommended 4-size maximum.

Why 16px prose body (not 14px): the lesson view is a reading experience, not a UI chrome element. Research consistently shows 16px as minimum comfortable reading size for long-form text. 14px is appropriate for sidebar labels and UI controls; it is too small for 400–600 word lesson bodies on a workstation monitor at arm's length.

Why 1.7 line height for prose: compliance technical content includes long command names and control IDs inline. 1.7 gives each line visual separation without making the column feel sparse.

Inline code (`<code>` inside prose): background `rgba(48,54,61,0.6)`, border-radius 3px, padding `var(--inline-code-padding-y) var(--spacing-xs)` (2px vertical / 4px horizontal — see spacing exceptions table), font `--font-mono`, size `--text-mono` (13px). This is distinct from a fenced code block — it is an inline annotation.

Declared weights for this phase: still exactly 2 — 400 (regular) and 600 (semibold). No new weights added.

- Position: right side of header bar (via flexbox justify-content: space-between)
- Size: 28px × 28px hit target (smaller than sidebar toggle — inline UI, not primary action)

- Hover state: background `rgba(255,255,255,0.06)`, icon color `--color-text-primary`
- Cursor: pointer
- `aria-label="Copy code to clipboard"`

**Copy button success state (2-second feedback):**
- Icon swaps: `Copy` → `Check` (both Lucide icons, same 14px size)
- Icon color changes: `#8b949e` → `#4ade80` (green-400 — success signal, distinct from accent orange)
- No toast notification — icon swap is the entire feedback mechanism (per CONTEXT.md specifics)
- After 2 seconds: icon reverts to `Copy`, color reverts to `#8b949e`
- Implementation: `setTimeout(() => { ... }, 2000)` in click handler
- `aria-label` updates during success: "Copy code to clipboard" → "Copied!" (reverts after 2s)

**Token area (`.code-block-body`):**
- Background: `--color-code-bg` (`#0d1117`) — Shiki sets this via `github-dark` theme
- Padding: `--spacing-md` (16px) top/bottom, `var(--code-block-padding-x)` (20px) left/right
- Overflow-x: auto (horizontal scroll for long lines — never wrap code)
- Font: `--font-mono` (JetBrains Mono)
- Font size: `--text-mono` (13px)
- Line height: 1.6

**Shiki configuration:**
- Theme: `github-dark` (matches `--color-code-bg` value exactly — no color conflict)
- Language: `powershell` (maps to PS 5.1 grammar)
- Shiki runs at content-load time (lazy, client-side) via the marked.js custom renderer
- The renderer calls `codeToHtml(code, { lang: 'powershell', theme: 'github-dark' })` asynchronously
- For non-PS blocks (e.g., batch, json, yaml), use the same theme with appropriate language identifier
- If Shiki grammar fails, fall back to `<pre><code>` with no highlighting — never throw

### 9. OT Callout Block

Source: D-03 (LOCKED), CONTEXT.md specifics section.

Authored in Markdown as:

```markdown
> [!OT]
> Body text here. Describe the OT-specific consideration.
```

The custom marked.js renderer detects `> [!OT]` blockquotes and converts them to the callout component. Standard blockquotes (without the `[!OT]` token) render normally.

**Visual anatomy:**

```
┌─────────────────────────────────────────────────────────────────────┐
│ ▌  IN OT ENVIRONMENTS                                               │
│ ▌                                                                   │
│ ▌  Body text of the OT-specific consideration. Can be multiple      │
│ ▌  sentences. References real pipeline OT constraints.              │
└─────────────────────────────────────────────────────────────────────┘
  ^
  3px left border, color --color-ot-border (#d97706)
```

**Container (`.ot-callout`):**
- Background: `--color-ot-bg` (`rgba(217,119,6,0.08)`)
- Border left: `3px solid --color-ot-border` (`#d97706`)
- Border radius: 0 4px 4px 0 (left edge is flush, right corners rounded)
- Padding: `--spacing-md` (16px) all sides
- Margin: `--spacing-lg` (24px) top and bottom (more breathing room than paragraph margin)

**Label (`.ot-callout-label`):**
- Content: "IN OT ENVIRONMENTS" — exact string, always uppercase, always this wording
- Font: `--font-mono` (JetBrains Mono) — monospace label creates a "stamp" visual effect appropriate for a technical callout
- Size: `--text-prose-small` (13px)
- Weight: 600 (semibold)
- Color: `--color-ot-label` (`#fbbf24` amber-400) — brighter than the border, provides contrast on the dark tint bg
- Letter spacing: 0.1em
- Margin bottom: `--spacing-xs` (4px)
- No icon — the label text alone is sufficient; adding a warning/info icon risks conflating the callout type with generic warnings

**Body text:**
- Font: Inter (var(--font-sans))
- Size: `--text-prose-body` (16px) — matches surrounding prose
- Color: `--color-ot-body` (`#e5e7eb` gray-200) — very slightly dimmer than `--color-text-primary` (`#f5f5f4`); callout reads as contextual supplement, not equal-weight body
- Line height: 1.6

**Standard blockquote (non-OT):**
For `>` blockquotes that do NOT begin with `[!OT]`, apply a simpler style:
- Border left: `3px solid --color-border` (`#3a3a3a`)
- Background: none
- Padding left: `--spacing-md` (16px)
- Text: `--color-text-muted` (`#737373`), italic

### 12. Loading State

Source: Claude's discretion (CONTEXT.md).

Shown while `fetch()` retrieves the lesson `.md` file. Targeted at < 500ms on GitHub Pages CDN — loading state must feel intentional, not like a flash.

**Loading indicator (`.lesson-loading`):**
- Replace lesson body content area with a pulse animation skeleton
- Three placeholder lines simulating a paragraph:
  - Line 1: 90% width, 16px height, border-radius 4px
  - Line 2: 75% width, 16px height, border-radius 4px
  - Line 3: 55% width, 16px height, border-radius 4px
- Gap between lines: `--spacing-sm` (8px)
- Animation: CSS `@keyframes pulse` — `opacity: 0.4 → 0.7 → 0.4`, 1.5s infinite ease-in-out (no shimmer gradient — avoids GPU layer promotion on many simultaneous elements)
- Skeleton line background: `--color-bg-secondary` (`#2a2a2a`)
- Shown immediately on route change, removed when `.lesson-body` innerHTML is set

**Loading label:** None — the skeleton conveys "loading" without text. Screen reader: add `aria-busy="true"` on `.lesson-wrapper` during fetch, remove when done. Add `aria-live="polite"` region with text "Loading lesson..." that clears on completion.

## Interaction Contracts

### Lesson Navigation (Sidebar)

- On `renderLesson({ moduleId, lessonId })` call: `setActiveLesson(moduleId, lessonId)` updates sidebar
- Previously active module remains expanded (do not collapse parent module when navigating lessons)
- Active lesson indicator: 3px left border `--color-accent`, background `rgba(249,115,22,0.08)`, font-weight 600
- Non-active available lessons: hover shows `rgba(249,115,22,0.04)` bg + accent text color
- Unavailable lessons: no hover, no cursor, opacity 40%
- Sidebar activation check: on `initSidebar()`, HEAD fetch each lesson URL; on 200 → render as `<a>`; on 404 → render as `<span aria-disabled="true">`

### Copy to Clipboard

- Click `.code-copy-btn`: call `navigator.clipboard.writeText(rawCodeText)`
- `rawCodeText` = the raw code string before Shiki processes it — store on the wrapper element as a `data-code` attribute (HTML-escaped) during rendering
- On success: swap icon `Copy` → `Check`, color `#8b949e` → `#4ade80`, `aria-label` → "Copied!"
- After 2000ms: revert icon, color, aria-label
- On `navigator.clipboard` failure (insecure context, permission denied): no visual change; log warning to console; do NOT show an error toast (clipboard is optional convenience, not critical path)
- Keyboard: button is focusable (tabIndex 0, receives focus ring); Enter and Space trigger click

### Lesson Prev/Next

- Both are `<a>` tags with `href="#/lesson/{moduleId}/{lessonId}"` — no click handler needed
- Disabled direction (first/last lesson): empty spacer `<div>` rendered instead of `<a>`
- Hover: border-color transitions to accent (150ms ease)
- Focus: standard focus ring (`3px solid #f97316`)

### Markdown Rendering Pipeline

Order of operations on route change to `#/lesson/{moduleId}/{lessonId}`:

1. Render loading skeleton immediately (synchronous DOM write)
2. Fetch lesson `.md` file from `{BASE_URL}data/modules/{moduleId}/lessons/{lessonId}.md`
3. Parse frontmatter (gray-matter or manual YAML split) — extract `title`, `complianceTags[]`, `complianceControls[]`
4. Call `marked.parse(markdownBody)` with custom renderer (OT callout + code block wrapper)
5. For each fenced code block detected: call Shiki `codeToHtml(code, { lang, theme: 'github-dark' })`
6. Inject rendered HTML into `.lesson-body`
7. Remove loading skeleton
8. Render compliance badge row from `complianceTags[]` via `renderBadge()`
9. Call `setActiveLesson(moduleId, lessonId)` on sidebar
10. Render prev/next footer from lesson order in `MODULES` config

## Accessibility Contract (Phase 2 additions)

Inherits all Phase 1 accessibility contracts. Phase 2 adds:

- Lesson `<article>` element wraps `.lesson-body` — semantic landmark for screen reader navigation
- Lesson `<h1>` is the lesson title — one h1 per page, always
- Code blocks: `<pre>` + `<code role="region" aria-label="PowerShell code block">` — makes code navigable as a region
- Copy button: `aria-label="Copy code to clipboard"` updates to `aria-label="Copied!"` on success; never empty
- OT callout: rendered as `<aside aria-label="OT environment note">` — distinguishes from main prose flow
- Compliance bar: `<div role="list" aria-label="Compliance controls covered">` with individual `<div role="listitem">` badges
- Loading state: `aria-busy="true"` on `.lesson-wrapper`; `<div aria-live="polite" class="sr-only">Loading lesson…</div>` clears to empty on load
- Error state: `role="alert"` on `.lesson-error` — screen reader announces immediately
- Lesson footer `<nav aria-label="Lesson navigation">` wrapping prev/next links
- Active lesson in sidebar: `aria-current="page"` on the `<a>` element
- All color contrast for new elements:
  - OT label `#fbbf24` (amber-400) on `rgba(217,119,6,0.08)` bg over `#1a1a1a` base: effective contrast ≈ 11:1 (AAA)
  - Code header text `#8b949e` on `#161b22`: contrast ≈ 4.8:1 (AA pass at 13px)
  - Copy success icon `#4ade80` (green-400) on `#161b22`: contrast ≈ 9.1:1 (AAA)
  - Prose body `#f5f5f4` on `#1a1a1a`: ≈ 17:1 (AAA) — same as Phase 1 body text
