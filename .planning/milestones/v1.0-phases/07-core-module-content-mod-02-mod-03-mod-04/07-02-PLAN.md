**FILE: quizzes/01.json**
id: "account-access-quiz-01"
moduleId: "account-access"
title: "Account & Access Management Knowledge Check"
Exactly 3 questions (q-01, q-02, q-03). Each has 4 answers (a, b, c, d). Exactly 1 correct per question.

q-01 (complianceControls: ["NIST-AC-2"]):
stem: "Which PowerShell cmdlet retrieves the current members of the Domain Admins group in Active Directory?"
a: "Get-ADUser -Filter * | Where-Object {$_.Groups -contains 'Domain Admins'}" (incorrect — Get-ADUser does not have a -Groups filter property and this syntax will not return group membership correctly; use Get-ADGroupMember targeting the group directly)
b: "Get-LocalGroupMember -Group 'Domain Admins'" (incorrect — Get-LocalGroupMember retrieves members of LOCAL groups on the current machine, not domain groups; Domain Admins is a domain group requiring Get-ADGroupMember)
c: "Get-ADGroupMember -Identity 'Domain Admins'" (correct — directly retrieves all member objects of the specified AD group; pipe to Select-Object Name, SamAccountName, Enabled for a compliance-friendly output)
d: "Get-ADGroup -Identity 'Domain Admins' -Properties Members" (incorrect — Get-ADGroup with -Properties Members returns a list of distinguished names (DNs), not the user objects with Name and Enabled status that are needed for an access review)
explanation: "Get-ADGroupMember -Identity 'Domain Admins' returns the member objects (users, groups, computers) with their name and account type. Pipe to Select-Object to control output fields. This is the correct PS 5.1 approach for NIST AC-2 account management reviews."

q-02 (complianceControls: ["NIST-AC-6"]):
stem: "During a quarterly access review, you find svc-historian in the Domain Admins group. What is the correct action?"
a: "Leave it — historian service accounts require Domain Admin rights to write data" (incorrect — historian service accounts do not require Domain Admin rights; they require permissions on the specific shares and databases they write to; Domain Admin is excessive and violates least privilege — NIST AC-6)
b: "Remove svc-historian from Domain Admins immediately without investigation" (incorrect — removal before investigation is the correct operational priority, but 'without investigation' is wrong because you need to understand how the account gained Domain Admin rights to determine whether other accounts are similarly compromised)
c: "Investigate how svc-historian gained Domain Admin membership, then remove it and restrict its permissions to only what is required for historian operation" (correct — investigation first establishes whether this is an isolated misconfiguration or part of a larger unauthorized change; removal and restriction follow investigation; this sequence satisfies both NIST AC-6 least privilege requirements and the forensic preservation principle)
d: "Disable svc-historian immediately to prevent any further unauthorized access" (incorrect — disabling the historian service account will immediately stop historian data collection, potentially disrupting OT visibility; investigate and remove from Domain Admins rather than disabling the account)
explanation: "Least privilege (NIST AC-6) requires that svc-historian holds only the minimum permissions needed for historian operation — never Domain Admin. Correct response: investigate the escalation path, then remove from Domain Admins and scope permissions properly. Disabling the account is too disruptive; blind removal without investigation loses forensic context."

q-03 (complianceControls: ["TSA-AccessControl", "NIST-AC-2"]):
stem: "Which PowerShell cmdlet finds all accounts in Active Directory that have a ServicePrincipalName (SPN) — identifying service accounts?"
a: "Get-ADServiceAccount -Filter *" (incorrect — Get-ADServiceAccount retrieves Managed Service Accounts (MSAs) and Group Managed Service Accounts (gMSAs), not regular user accounts that happen to have SPNs registered; use Get-ADUser with an SPN filter for standard service accounts)
b: "Get-ADUser -Filter {ServicePrincipalName -like '*'} -Properties ServicePrincipalName" (correct — filters user accounts that have any SPN registered; the -Properties ServicePrincipalName flag is required because SPN is not returned by default; pipe to Select-Object Name, Enabled, ServicePrincipalName for a compliance-ready list)
c: "Get-Service | Where-Object {$_.StartType -eq 'Automatic'}" (incorrect — Get-Service retrieves Windows service configuration on the local machine; it does not query Active Directory for service account SPNs)
d: "Get-ADUser -Filter {Enabled -eq $true} -Properties *" (incorrect — returning all properties for all enabled users is wasteful and does not filter for SPNs specifically; use the ServicePrincipalName filter to target only service accounts)
explanation: "Get-ADUser -Filter {ServicePrincipalName -like '*'} -Properties ServicePrincipalName identifies all domain user accounts with registered Kerberos SPNs — these are your service accounts. This query supports NIST AC-2 account management by giving you a complete inventory of accounts that run services and may hold elevated permissions."
