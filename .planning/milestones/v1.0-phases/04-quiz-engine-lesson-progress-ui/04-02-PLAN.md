## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| quiz JSON → innerHTML | Static authored JSON fetched from public/data/; content inserted into DOM via innerHTML |
| answer click → progressStore | User click triggers score recording; event delegation validates click target before processing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-W1-01 | Tampering | quiz-engine.js buildFirstVisitHtml / buildRevisitHtml | mitigate | esc() applied to ALL quiz JSON strings before innerHTML insertion: quiz.title, question.stem, answer.text, answer.feedback, question.explanation — no exception |
| T-04-W1-02 | Tampering | quiz-engine.js JSON parsing | mitigate | Only access expected fields (id, title, questions, stem, answers, explanation, complianceControls) — never use JSON.parse result keys as dynamic property names |
| T-04-W1-03 | Tampering | quiz-engine.js quiz fetch URL | accept | quizId comes from lesson frontmatter (trusted authored content), not from URL hash or user input. The fetch URL is import.meta.env.BASE_URL + static path + authored quizId value |
| T-04-W1-04 | Tampering | quiz-engine.js answer click handler | mitigate | Event delegation guards: e.target.closest('.quiz-answer-btn'); questionCard.dataset.answered === 'true' check prevents double-submission; pointer-events:none set on all answers after first click |
| T-04-W1-05 | Information Disclosure | quiz JSON correct answer fields | accept | data-correct attribute is set on answer buttons in DOM — the "correct" answer is visible in DevTools. This is acceptable for a training app with no adversarial use case; cheating via DevTools is not a threat model concern |
</threat_model>
