## The Role of Patch Policy in Compliance

A patch management policy defines what must be patched, by when, and what documentation is required when patching is deferred. Without a written policy, an auditor cannot verify that the organization's patch decisions were intentional rather than accidental — and inconsistent or undocumented patch decisions are themselves an audit finding, separate from the underlying vulnerability.

Under the current TSA pipeline security directive, pipeline operators must have a documented patch management policy that covers four elements:

1. **Asset scope** — Identifies all IT and OT assets subject to the patch management program. The scope must explicitly include OT systems (historians, HMIs, engineering workstations, PLCs where applicable) — not just IT infrastructure.
2. **Risk-based timelines** — Defines the required patch timeline for each severity tier (e.g., critical CVEs patched within 30 days on IT systems; OEM-qualified timeline for OT systems). Timelines must be consistent and documented before an audit occurs, not constructed after a finding.
3. **Compensating controls process** — Defines what must happen when a patch cannot be applied on the standard timeline: who documents the compensating control, who approves it, at what interval it is reviewed, and what triggers its retirement.
4. **Responsibility assignment** — Identifies by role who is responsible for patch approval, deployment, and documentation sign-off. "The IT team" is not sufficient; a named role (e.g., IT Security Manager, OT Systems Engineer) must be assigned.

## Compensating Controls Documentation

When a patch cannot be applied on the standard timeline — most commonly for OT/ICS systems awaiting OEM qualification — the organization must document a compensating control. A compensating control is a security measure that reduces the risk of the unpatched vulnerability while the patch qualification cycle completes.

For a historian server like PIPELINE-HIST01 with an unpatched OS vulnerability, appropriate compensating controls include:

- **Network isolation** — Block all inbound connections to PIPELINE-HIST01 except those required for historian polling (specific source IPs and ports only). Implement via firewall policy and document the rule set as evidence.
- **Enhanced monitoring** — Configure SIEM alerting on any new or anomalous connection attempts to the historian IP. Alert criteria must be documented.
- **Restricted physical access** — Restrict physical access to the server room housing PIPELINE-HIST01 to named individuals. Document the access roster.

Each compensating control must meet four requirements:

1. **Documented in writing** — specifying the vulnerability it mitigates (CVE number and description), the control implemented, and the implementation date.
2. **Approved by a named responsible person** — not a team or group; a specific individual who accepts accountability for the risk decision.
3. **Reviewed at a defined interval** — minimum every 30 days; the review must be logged with date and reviewer name.
4. **Retired when the patch is applied** — the compensating control record must be closed out with the patch deployment date and KB number. Open compensating control records after a patch has been applied are a separate finding.

After deploying patches to generate the audit evidence file, export the full inventory:

```powershell
# TSA-PatchMgmt: Export full patch inventory for audit submission — run on the target system
Get-Hotfix | Export-Csv 'C:\Audit\patch-status.csv' -NoTypeInformation
```

This CSV is the primary evidence file submitted to auditors. It must be dated — use the file creation date in metadata, or include a date stamp in the filename (e.g., `patch-status-2025-10-15.csv`). Store the file securely and reference it by filename in the compensating controls documentation for any deferred patches. The CSV links the installed KB inventory to the compensating controls record: auditors can verify that patches listed as deferred are not present in the installed KB list.

## Audit Evidence Requirements

TSA auditors reviewing a pipeline operator's patch management program will request the following evidence categories:

- **Patch inventory report** — Generated by `Get-Hotfix | Export-Csv` (Exercise 1 in Lesson 2). Must cover all in-scope systems, not just domain-joined IT machines.
- **Compensating controls log** — A record for every deferred OT patch, including vulnerability ID, compensating controls implemented, approval date, and review dates. This is the most common missing evidence item in TSA patch management audits.
- **Change management records** — For each patch deployment: who approved it, when it was deployed, on which system, and the change window reference number.
- **OEM vendor qualification references** — For every OT patch, the OEM vendor communication or published advisory confirming the patch has been qualified for the specific software version installed.

To verify that the audit evidence file was created and contains usable data before submitting:

```powershell
# NIST MA-2: Verify the audit evidence file was created and contains records
Get-Content 'C:\Audit\patch-status.csv' | Measure-Object -Line
```

If the line count is `1` (header row only), the export ran but captured no installed patches — investigate before submitting this file to auditors. A patch inventory with only a header row signals either a permissions issue on the target system or that the system's WMI patch database is not populating correctly. A missing file or a zero-line-count result is a finding on its own.

## Policy Timelines and Escalation

The following table illustrates representative patch policy timelines. Organizations must define their own timelines in their patch management policy and apply them consistently — inconsistent application (e.g., patching some critical CVEs within 30 days and others after 90 days with no documented reason) is itself an audit finding.

| Scenario | Required Timeline | Documentation Required |
|----------|------------------|----------------------|
| Critical CVE on IT system (CVSS 9.0+) | Patch within 30 days | Change record with approver name and date |
| High CVE on IT system (CVSS 7.0-8.9) | Patch within 60 days | Change record |
| Any CVE on OT system | Await OEM qualification (3-12 months typical) | Compensating controls doc + OEM qualification reference |
| Patch fails to install during maintenance window | Reschedule within next available window | Failure log with error details; OEM vendor notified |
| Missed patch window with no compensating control | Immediate escalation to CISO | Incident report; TSA notification required if OT-impacting |

The "missed patch window with no compensating control" row is the scenario that triggers TSA notification obligations. If an OT system passes its scheduled maintenance window without the patch being applied, and no compensating control has been documented and approved, the organization has a gap that may require reporting to TSA depending on the vulnerability severity and the system's role in pipeline operations.

> [!OT]
> In OT environments — a deferred patch with no documented compensating control is the single most common patch management finding in TSA pipeline security audits. The absence of a compensating control document is treated as a compliance gap regardless of whether the patch was technically safe to defer. Before a maintenance window closes, ensure the compensating controls document for every deferred OT patch is signed, dated, and stored in the compliance records system — not just in a personal email thread. If your organization uses a ticketing system (ServiceNow, Jira, etc.), link the compensating control record to the patch ticket so the evidence chain is unbroken.
